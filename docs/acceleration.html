Acceleration Architecture

Hardware acceleration should keep the current CPU implementation as the correctness reference while adding CUDA and Metal backends behind a shared operation layer.

Why Add It

The assistant baseline showed that useful experiments need more training throughput. Re-running the same CPU loop is too slow for model-size and step-count sweeps.

Current Bottlenecks

  • SeqTensor and Parameter own host-only std::vector<float> storage.
  • Attention softmax/value accumulation, loss, and gradient clipping are still scalar host loops.
  • Forward and backward passes allocate many temporary vectors every step.
  • Evaluation repeats full forward passes on CPU batches.
  • Generation repeatedly recomputes the full context without a key/value cache.

Backend Shape

Add a backend abstraction first, then migrate operations behind it. The public training and generation commands should select a backend without changing checkpoint semantics. 

./bin/mgpt train --backend cpu
./bin/mgpt train --backend metal
./bin/mgpt train --backend cuda

Build-time library selection should be automatic. On macOS, make BACKEND=metal links Metal, Foundation, and QuartzCore frameworks. Use make deps to check local backend availability. Requesting BACKEND=metal on non-macOS platforms fails at build time instead of silently compiling the CPU stub. 

make BACKEND=metal all
./bin/mgpt backends

Metal can now be selected and probed at runtime. Linear forward and backward, feed-forward, LayerNorm, and AdamW have Metal paths. CPU remains the reference backend, and unsupported operations still fall back to CPU behavior.

Local Backend Support

The current development machine is an Apple Mac, so Metal is the only hardware backend that can be tested locally. CUDA should remain in the architecture, but it needs CI or a separate NVIDIA machine before it can be treated as supported.

If metallib is unavailable, the first Metal backend should use runtime source compilation through the Metal framework rather than requiring an offline metallib build step.

Implementation Order

  1. Add backend selection, capability reporting, and CPU parity tests.
  2. Introduce backend-owned tensor buffers with explicit host/device synchronization.
  3. Move linear forward and backward to backend operations first.
  4. Move general matrix multiply operations behind the same interface.
  5. Add backend layernorm, GELU, softmax attention, loss, gradient clipping, and AdamW.
  6. Add CUDA and Metal implementations once the operation interface is stable.
  7. Add benchmarks that report tokens/sec, steps/sec, memory use, and eval runtime per backend.

Linear forward/backward and GELU forward/backward now route through the backend operation interface and have lab parity coverage. First Metal kernels exist for those operations and fall back to CPU when Metal is unavailable.

Persistent Metal buffer primitives are available through BackendBuffer, with lab coverage for upload/download round trips. Linear layers now reuse cached Metal buffers across calls and avoid re-uploading forward input and weights for the matching backward pass. The full model does not yet keep every activation and parameter resident across whole training steps.

Benchmarking

Use bench to compare backend selection, model config, training throughput, and generation throughput. 

./bin/mgpt bench

make BACKEND=metal all
./bin/mgpt bench --backend metal --train-steps 20 --gen-tokens 20

./bin/mgpt bench --backend metal --resident-linear --iterations 1000 --rows 128 --in 128 --out 128
./bin/mgpt bench --backend cpu --resident-linear --iterations 1000 --rows 128 --in 128 --out 128

The resident linear benchmark uploads inputs and parameters once, batches repeated linear forward/backward work into a single Metal command buffer, then downloads final outputs and gradients once. It is intended to measure Metal kernel and buffer behavior without the full training loop moving data between every operation.

Benchmark output includes backend_uploads, backend_upload_bytes, backend_downloads, and backend_download_bytes. Use those fields to separate raw kernel performance from host/device synchronization overhead.

Feed-forward execution also has fused Metal paths for linear -> GELU -> linear forward work and the matching backward gradient work. These paths batch adjacent kernels into one command buffer, and the intermediate activations stay in shared backend buffers across the forward/backward boundary. AdamW updates now batch all parameter work into a single Metal submission before results are downloaded. Full training is not resident end to end yet.

Correctness Rules

  • The CPU backend remains the reference implementation.
  • Each accelerated operation needs CPU parity tests with tolerances.
  • Metal-trained checkpoints must load on CPU and produce matching logits and greedy generation within documented tolerances.
  • Training should produce comparable loss curves for the same seed, config, and data.
  • Checkpoints should remain portable across backends.
  • Data parsing, tokenization, checkpoint metadata, and eval reports stay backend-independent.

The lab includes cpu_metal_training_parity_test, backend_adamw_update_test, and metal_checkpoint_cpu_generation_interop_test. These are no-ops on CPU-only machines and active regression tests when a Metal runtime device is visible.

The lab also includes backend_gelu_op_test, backend_layernorm_op_test, backend_feedforward_forward_test, backend_feedforward_backward_test, backend_adamw_batch_submission_test to verify the optimizer batching path, and gradient_check_feedforward to verify activation, normalization, and fused feed-forward backend parity while catching feed-forward activation derivative regressions.

A staged parity trace test now compares CPU and Metal at the embedding, block, logits, gradient, and AdamW update stages and prints the first divergent stage when parity breaks.
