How GPT Uses Tokens

A GPT-style model does not read text as words or characters directly. It reads a stream of tokens, turns them into vectors, processes those vectors through transformer blocks, and predicts the next token at each step.

That matters because every setting changes the token stream or the model's ability to use it. If the tokenizer splits text into many small pieces, the model sees longer sequences. If the context window is too short, the model cannot use older text even if it is present in the chat history. If the model is too small, it can fail to learn the patterns in the tokens it does see.

Context Window

What it is

--context sets the number of tokens the model can process in one forward pass. It is the size of the working memory the checkpoint is built around.

If it is too small

The model cannot see enough of the prompt, prior turns, or training example. In chat, older turns get trimmed earlier. In training, the model only learns short-range dependencies, so it may fail on tasks that need the full instruction, a format marker, or a longer answer.

If it is too large

Memory use rises because attention scales with sequence length. The model also needs more data to make good use of the extra window. On tiny datasets, a huge context can waste capacity on positions that are rarely meaningful.

Rule of thumb: choose the smallest context that still fits the real task. For arithmetic or short instruction tasks, a small window can be enough. For chat or session memory, use a larger window, but only if you have enough training data to fill it.

Tokenizers

Byte tokenizer

Byte mode maps each byte directly into the token stream. It is simple, robust, and easy to debug. Every input can be represented exactly, including unusual punctuation or malformed text.

Tradeoff: common words take more tokens, so sequences are longer and the context window fills up faster.

BPE tokenizer

BPE merges frequent character sequences into single tokens. That usually shortens text, which means the same context window can hold more semantic content. It often learns natural language and assistant-style instruction data more efficiently.

Tradeoff: the token boundaries are less obvious when debugging, and you need the checkpoint to carry the tokenizer choice with it.

When to choose which

Use byte when you want the simplest pipeline, when you are validating a new feature, or when the text may be arbitrary. Use bpe when you want better efficiency on natural language, more useful context usage, and a better fit for assistant prompts.

Model Capacity

Setting Behavior What changes when you increase it What can go wrong if it is too large
--d-model Hidden width of the transformer. More representational space for token and position features. Higher compute cost and more chance of overfitting on small data.
--layers Number of stacked transformer blocks. More rounds of transformation and abstraction. Slower training, more parameters, and more sensitivity to data quality.
--heads Attention heads per block. More subspaces for the model to track different relationships in parallel. Must divide d-model; too many heads with too little width can make each head weak.
--ff Feed-forward expansion width. More non-linear capacity inside each block. Extra compute for gains that may not help if the dataset is tiny.

Width and depth are the main capacity knobs. Width gives the model more room to represent features. Depth gives it more steps to combine those features into useful behavior. On small datasets, the wrong answer is not always "make it bigger"; often the better move is to keep the model modest and make the data cleaner and the prompts more consistent.

Training Knobs

--batch-size

Batch size controls how many context windows are sampled per update. Bigger batches usually make gradients less noisy, but they need more memory. For a tiny model, very large batches can make training less responsive to new information.

--lr

Learning rate controls how aggressively the optimizer updates the weights. If it is too high, training can become unstable or bounce around. If it is too low, the model may appear to do nothing.

--tokenizer

Tokenization affects the training sequence length, how often the model sees repeated fragments, and how much of the context window is spent on separators versus content.

Training settings do not directly change generation randomness, but they strongly shape the checkpoint you end up with. A good training run on a bad configuration still produces a poor checkpoint. A modest model trained on the right shape of data often beats a larger model trained with mismatched settings.

Generation Knobs

--temperature

Lower temperature concentrates probability mass on the most likely tokens. That makes output more repeatable and often more accurate for exact-answer tasks. Higher temperature increases variety, which can be useful for brainstorming but also introduces more mistakes.

--top-k

Top-k limits sampling to the k most likely next tokens. Smaller values make output more conservative. 1 is greedy decoding, which is usually best for exact-match evaluations and controlled assistant baselines.

Prompt length

Generation uses the checkpoint context window. If the prompt or conversation is longer than that window, the oldest part is trimmed. That means the model only conditions on the most recent tokens it can fit.

For evaluation, keep sampling conservative. If you are trying to see whether the model actually learned a dataset, greedy or near-greedy decoding gives the cleanest signal.

Practical Recipes

Debugging the pipeline

  • Use byte tokenizer.
  • Keep context modest.
  • Keep the model small.
  • Use greedy generation.

Tiny assistant data

  • Prefer bpe if the data is natural-language heavy.
  • Keep context large enough to hold the full prompt plus recent history.
  • Do not inflate width or depth before the model can memorize the dataset shape.

Broadening later

  • Increase context only when the task actually needs longer memory.
  • Increase model width or depth only when the current model is clearly underfitting.
  • Relax sampling only after the model is already producing correct answers.
