Code Guide

Framework Guide

This page is the code-side guide for the framework. It covers the supported entry points, the lower-level helpers, and the chat controls you can call from application code.

What To Use

C++ facade

  • microgpt::api::load_model to open a checkpoint.
  • microgpt::api::generate to run inference.
  • microgpt::api::train_text_file to train from raw text.
  • microgpt::api::ModelHandle to keep model state together.

C facade

  • microgpt_load_model to load a checkpoint.
  • microgpt_generate_text to generate text.
  • microgpt_free_model and microgpt_free_string for cleanup.
  • microgpt_last_error for exception-free error reporting.

C++ API Details

Load a model

Use microgpt::api::load_model(checkpoint_path, backend) when you already have a checkpoint. It returns a ModelHandle that owns the model, optimizer state, and step number.

Generate text

Call microgpt::api::generate(handle, prompt, options). The options are: max_new_tokens, temperature, top_k, and stop_token_id. Generation uses the model's tokenizer and backend already stored in the handle.

Train from text

Call microgpt::api::train_text_file(input_path, checkpoint_path, options) to build a checkpoint from a plain text corpus. The train options let you control model size, context length, batch size, learning rate, backend, and whether the validation split is enabled.

Training Options

  • steps: number of optimization steps.
  • context_length: model context window used for training.
  • d_model, num_layers, num_heads, d_ff: model size controls.
  • batch_size: training batch size.
  • learning_rate: optimizer learning rate.
  • seed: random seed for reproducibility.
  • backend: CPU, Metal, or CUDA backend selection.
  • use_validation_split: turn the held-out split on or off.
  • validation_split_ratio: fraction used for the training side when splitting is enabled.
If you want to train and evaluate on all of the available data, set use_validation_split = false. That keeps the full token set in play instead of partitioning it into train and held-out slices.

Generation Options

  • max_new_tokens: number of tokens to generate.
  • temperature: sampling temperature.
  • top_k: top-k sampling cutoff.
  • stop_token_id: token ID that ends generation, usually Tokenizer::kEos.

Direct Use Example

#include "microgpt/api.hpp"

int main() {
  auto model = microgpt::api::load_model("model.bin");
  microgpt::api::GenerationOptions options;
  options.max_new_tokens = 16;
  options.top_k = 1;
  std::string text = microgpt::api::generate(model, "Hello", options);
}

Training Example

#include "microgpt/api.hpp"

int main() {
  microgpt::api::TrainingOptions options;
  options.steps = 200;
  options.context_length = 64;
  options.use_validation_split = false;
  auto model = microgpt::api::train_text_file("train.txt", "model.bin", options);
}

Lower-Level Framework Calls

If you want direct access to the framework internals, these are the main helpers.

  • generate_text(Model&, prompt, max_new_tokens, temperature, top_k, stop_token_id, on_progress) for raw generation.
  • train_model(Model&, train_tokens, val_tokens, AdamW&, steps, checkpoint_path, resume_step, on_progress) for optimization loops.
  • save_checkpoint(path, model, opt, step) and load_checkpoint(path, opt, step) for persistence.
  • split_train_val(tokens, train, ratio) if you want manual train/validation slicing.
  • write_instruction_examples, parse_instruction_examples, write_session_examples, and parse_session_examples for dataset work.
  • validate_instruction_text and validate_session_text for contract checks.

Chat Helpers

The chat layer is a set of reusable helpers, not just a UI. These are the functions used by the CLI and TUI to manage context.

  • format_multi_turn_chat_prompt(history, user_prompt) builds the model prompt from prior turns.
  • trim_chat_history_to_context(history, user_prompt, context_length, tok) removes old turns until the prompt fits.
  • compress_chat_history(history, tok, context_length) explicitly compresses long histories.
  • summarize_chat_session(history, tok) reports message and token counts.
  • chat_commands() exposes the supported slash commands.
Chat command behavior:
  • /reset resets the whole conversation state.
  • /clear clears the active context but keeps the transcript on screen.
  • /compress compresses the current context on user request.

C API

The C API exposes an opaque microgpt_model, checkpoint loading, generation, owned string cleanup, and microgpt_last_error for exception-free error reporting. 

#include "microgpt/api_c.h"

microgpt_model* model = microgpt_load_model("model.bin", "cpu");
microgpt_generation_options options = microgpt_default_generation_options();
char* text = microgpt_generate_text(model, "Hello", options);
microgpt_free_string(text);
microgpt_free_model(model);

The normal build creates bin/libmicrogpt.a and compiles examples/embed_c.c as bin/embed_c_example.

Entry Points By Task

Load and run inference

Use api::load_model or microgpt_load_model first, then call api::generate or microgpt_generate_text.

Train on a text file

Use api::train_text_file when you want the framework to handle tokenization, optional split control, training, and checkpoint saving in one call.

Build a custom training loop

Use train_model, split_train_val, save_checkpoint, and the tokenizer helpers directly if you need to plug the framework into another pipeline.

Add chat UX

Use the chat helpers to build a prompt, manage trimming, and trigger compression when the user explicitly requests it.

Build Notes

The build already compiles the embedding examples, so the public API and C API are part of the normal build path. The internal helpers are header-only and can be reused directly from source-level integrations.

Stability

This is a code guide, not an ABI guarantee. The C API is the safer boundary for external consumers, while the C++ facade is the easiest source-level integration point.
